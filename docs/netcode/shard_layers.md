# Shard Layers

## Назначение

Layer внутри шарда - это отдельная, изолированная runtime-сессия одной и той же игры. Шард остается верхнеуровневой единицей продукта, а layer решает проблему плотности: один слой не должен одновременно держать больше 10 активных участников.

## Базовая модель

- `shard` хранит метаданные мира: игру, seed, статус, владельца, общую конфигурацию.
- `layer` хранит живую симуляцию: состав участников, mob/resource/boss state, прогресс и активные bot runs.
- Один shard может иметь несколько layers.
- Один layer имеет жесткий cap: максимум 10 активных игроков.
- Активным считается участник, который участвует в симуляции этого layer в текущем окне присутствия.

## Default rules

Если пользователь входит в мир, система действует без дополнительных уточнений:

1. Сначала пытается поставить его в уже открытый layer с подходящей игрой и свободным местом.
2. Если есть несколько свободных layers, выбирается наименее заполненный.
3. Если есть группа с совпадающими prime-hours, приоритет у layer, где уже есть overlap этой группы.
4. Если подходящего layer нет, создается новый layer того же shard.

Это правило считается достаточным по умолчанию, поэтому отдельное уточнение от пользователя не требуется.

## Что значит "свободный layer"

- Это layer, где occupancy ниже 10.
- Свободный layer не обязан быть пустым.
- Пользователь может быть переведен на другой layer только через явное действие системы или при создании новой сессии.
- Автоперенос между layers во время одной короткой сессии лучше не делать без причины, чтобы не ломать ощущение наблюдаемого прогресса.

## Изоляция

Каждый layer имеет собственную симуляцию:

- свои мобы;
- свои resource nodes;
- свой boss HP pool;
- свой drop table;
- свой bot runtime.

Общий у shard только seed и правила генерации. Это важно, чтобы layers были предсказуемыми и не мешали друг другу.

## Ограничения

- Не стоит пытаться держать в одном layer больше 10 участников, даже если они в основном пассивные наблюдатели.
- Не стоит смешивать несколько независимых групп в один overloaded layer, если есть свободный.
- Не стоит требовать от пользователя вручную выбирать layer до того, как система научится автоматически раскладывать участников.
- Не стоит делать shared boss state на все layers сразу, иначе cap 10 теряет смысл.

## UX на фронте

- В профиле должен быть список shard tabs ниже prime grid.
- В карточке игры кнопка `Войти в мир` создает shard или добавляет пользователя в layer.
- Если layer переполнен, UI должен показывать, что будет создан новый layer, а не молча бросать ошибку.
- Для дебага полезно показывать occupancy, layer id и shard seed short.

## Когда можно расширять

Когда появится реальная нагрузка, можно добавить:

- ручной перенос между layers;
- soft queue при переполнении;
- отдельную политику для party / guild / overlap group;
- слой-only persistence для реплеев и отладки.

На первой версии этого не требуется.
